## 1. 프로젝트 개요

### 1.1 배경 및 목적

현재 국내 비영리단체(NPO)의 후원관리는 **MRM(휴먼소프트웨어), 도너스(DONUS)** 등의 독립 솔루션에 의존하고 있으며, ERP(예산회계시스템)와의 연동이 수동이거나 별도 API 연동 작업이 필요한 구조다.

본 시스템은 다음의 핵심 문제를 해결하기 위해 기획된다.

- **문제 1 — 이중 관리의 비효율**: CMS 회원, 계좌이체 회원, 웹 가입 회원을 별도로 관리하여 누락·오류 발생
- **문제 2 — ERP 연동 단절**: 후원금 수납 이후 ERP 수입결의 처리가 수동으로 진행되어 재정 투명성 저하
- **문제 3 — 캠페인 유연성 부족**: 기념품(굿즈) 연계, 캠페인별 개인화가 어려움
- **문제 4 — 후원자 여정 단절**: 최초 후원 신청부터 영수증 수령까지 일관된 디지털 경험 부재

### 1.2 핵심 목표

```
후원자가 편하게 후원하고,
단체가 효율적으로 관리하며,
ERP가 자동으로 정산되는 시스템
```

### 1.3 프레임워크 방법론 참고

**PwC Digital Transformation 5 Essentials** 기반:

| 요소 | 적용 |
|------|------|
| Vision & Strategy | NPO ERP 완전 연동, 후원자 여정 디지털화 |
| Platform Architecture | API-First, RESTful Open API 설계 |
| Processes & Ways of Working | 수납→결의→영수증 자동화 프로세스 |
| Governance & Operating Model | 본부/지부/담당자별 권한 분리 |
| Performance Management | 후원자 감소율(Churn Rate) 리포트, 캠페인 ROI 대시보드 |

**Deloitte Quick Wins 접근법** 적용:
- Phase 1: 캠페인 페이지 + 결제 연동 (빠른 성과)
- Phase 2: CRM + 자동화 (핵심 운영)
- Phase 3: ERP 완전 연동 + AI 분석 (전략적 확장)

## 2. 시장 현황 및 벤치마킹

### 2.1 국내 주요 경쟁 솔루션 분석

| 구분 | MRM (휴먼소프트웨어) | 도너스 (DONUS) | 도네이션박스 |
|------|---------------------|---------------|-------------|
| **형태** | 설치형 앱 (클라우드 DB) | 웹 SaaS | 웹 SaaS |
| **주요 강점** | 통합 인적자원 관리, 자동화 | ERP 수입결의 API 연동 | 온라인 모금함 |
| **결제(정기)** | CMS/카드/네이버페이/카카오페이/페이코/무통장 | 카드/CMS/계좌이체 | 카드/CMS/URL결제 |
| **결제(일시)** | 카드/실시간계좌이체/가상계좌/네이버페이/카카오페이/페이코 | 카드/계좌이체 | 카드/실시간이체 |
| **ERP 연동** | 수입결의서 자동처리 지원 | **REST API (List payments / Change status)** | 미지원 |
| **캠페인 페이지** | 무제한, 무료 | 제한적 | 무제한 |
| **핵심 한계** | 설치형 → 모바일 UX 취약 | CRM 기능 약함 | ERP 연동 없음 |

### 2.2 글로벌 벤치마킹

| 솔루션 | 벤치마킹 포인트 |
|--------|---------------|
| **Bloomerang** | 후원자 유지율(Donor Retention) 대시보드 → 후원자 감소 조기 경보 |
| **Salesforce NPSP** | 구성원 > 약정 > 납입의 계층적 데이터 모델, Household 관리 |
| **Funraise** | AI 기반 성과 예측 및 이상 감지 |
| **Donorbox** | 임베드 가능한 기부 위젯, 다국어/다통화 지원 |

### 2.3 도너스 ERP 연동 구조 분석 (참고 문서 기반)

도너스의 ERP 수입결의 연동은 다음 2개 API로 구성된다. 본 시스템도 동일한 패턴을 채택한다.

```
연동 흐름:
[후원자 결제] → [수입대기 상태로 납입 등록]
    → [ERP: 납입정보 조회 API (List payments)]
    → [ERP 내 수입결의서 생성]
    → [ERP: 납입상태 변경 API (Change status → 수입완료)]
```

**수입 상태값 4가지:**

| 상태 | 설명 |
|------|------|
| `수입대기` | 납입 등록 시 자동 지정되는 초기값 |
| `수입진행` | 수입완료 전 중간단계 처리 중 |
| `수입완료` | 수입결의가 완료된 납입 |
| `수입제외` | 수입으로 인식하지 않는 납입 |

## 3. 사용자 여정(User Journey) 분석

### 3.1 후원자 여정 (Donor Journey)

```
[인지] → [관심] → [첫 후원] → [지속 후원] → [이탈 방지] → [업그레이드]
```

| 단계 | 후원자 행동 | 시스템 역할 | 해결할 문제 |
|------|-----------|------------|------------|
| **1. 인지** | SNS/광고 클릭 | 랜딩페이지 노출 | GA/픽셀 트래킹으로 전환 측정 |
| **2. 관심** | 캠페인 페이지 탐색 | 캠페인 스토리, 모금 현황 표시 | 신뢰감 형성 |
| **3. 첫 후원** | 후원 신청 폼 작성 | 간편 결제 연동, 본인인증 | 이탈 최소화 (3클릭 완료) |
| **4. 확인** | 결제 완료 메시지 수신 | 자동 감사 SMS/이메일 발송 | 즉각적 피드백 |
| **5. 지속** | 마이페이지 이용 | 납부내역 조회, 정기후원 관리 | 투명한 정보 제공 |
| **6. 영수증** | 기부금 영수증 요청 | 자동 발급 / 국세청 연계 | 세금 혜택 제공 |
| **7. 이탈 위기** | 결제 실패 / 무응답 | 자동 미납 알림, 재결제 유도 | 이탈 방지 |
| **8. 업그레이드** | 추가 캠페인 참여 | 개인화 캠페인 추천 | LTV 향상 |

### 3.2 관리자 여정 (Admin Journey)

```
[캠페인 기획] → [페이지 생성] → [모금 운영] → [수납 처리] → [ERP 정산] → [성과 분석]
```

| 단계 | 관리자 행동 | 시스템 역할 |
|------|-----------|------------|
| 캠페인 기획 | 목표금액/기간/결제수단 설정 | 캠페인 CMS |
| 페이지 생성 | 이미지/문구/금액 설정 | 노코드 편집기 |
| 모금 운영 | 후원자 문의 응대 | 상담 이력 관리 |
| 수납 처리 | CMS 출금일 확인 | 자동 출금 + 미납 처리 |
| ERP 정산 | 수입결의 생성 | **API 자동 연동** |
| 성과 분석 | 캠페인별 ROI 확인 | 대시보드 리포트 |

## 4. 기능 요구사항 - 랜딩페이지 & 캠페인

### 4.1 캠페인 페이지 생성

**왜 필요한가**: 단체가 매번 개발팀에 의뢰하지 않고 직접 캠페인 페이지를 만들 수 있어야 한다. MRM의 무제한 캠페인 페이지 정책처럼, 운영 유연성이 핵심이다.

#### 필수 기능

- **노코드 편집기**: 드래그 앤 드롭으로 캠페인 페이지 생성
- **캠페인 템플릿**: 정기후원 / 일시후원 / 정기+일시 선택형 / 굿즈(기념품) 연계형
- **개인화 설정 항목**:
  - 배경이미지 / 완료이미지 / 썸네일 이미지
  - 캠페인 제목 및 안내 문구
  - 기부유형 (정기 / 일시 / 정기+일시 선택)
  - 기부금액 프리셋 (예: 1만원, 3만원, 5만원, 직접입력)
  - 결제수단별 사용 여부 (CMS / CMS+카드 / CMS+카드+간편결제 등 조합)
  - 추가 입력 정보 (추천인, 응원메시지 등 30여 개 커스텀 필드)
  - 콜백 URL 설정

#### 캠페인 페이지 추가 기능

- **모금 현황 위젯**: 실시간 달성률, 후원자 수, 모금액 표시
- **SNS 공유 버튼**: 카카오톡, 페이스북, 링크 복사
- **GA 추적 코드 삽입**: 캠페인별 Google Analytics 연동
- **픽셀 코드 삽입**: Meta/카카오/네이버 광고 픽셀 추가
- **임베드 위젯**: 외부 사이트에 후원 폼 삽입 가능 (iframe/JavaScript)
- **P2P 펀드레이징**: 후원자가 개인 모금 페이지를 생성하여 지인에게 공유

#### 예외 처리 및 정책

- 캠페인 종료일 이후 접속 시: "캠페인이 종료되었습니다" 안내 페이지 표시
- 목표 금액 초과 달성 시: 관리자 알림 발송, 초과 후원 허용/마감 정책 설정 가능
- 미리보기 모드: 공개 전 관리자만 접근 가능한 미리보기 URL 제공

## 5. 기능 요구사항 - 결제 연동

### 5.1 지원 결제 수단

**왜 필요한가**: 후원자 이탈의 주요 원인 중 하나는 원하는 결제 수단이 없는 것이다. 한국 후원자는 CMS 자동이체, 네이버페이, 카카오페이를 선호한다. (MRM FAQ 기반)

#### 정기후원 결제 수단

| 결제 수단 | 설명 | 우선순위 |
|----------|------|---------|
| **CMS 자동이체** | 금융결제원 오픈뱅킹 기반, 매월 자동 출금 | 필수 |
| **신용카드 정기결제** | 카드사 정기청구 | 필수 |
| **네이버페이** | 네이버페이 정기결제 | 높음 |
| **카카오페이** | 카카오페이 정기결제 | 높음 |
| **페이코** | PAYCO 정기결제 | 중간 |
| **무통장입금** | 가상계좌 발급 후 수동 확인 | 중간 |

#### 일시후원 결제 수단

| 결제 수단 | 설명 | 우선순위 |
|----------|------|---------|
| **신용카드** | 일반 카드 결제 | 필수 |
| **실시간 계좌이체** | 인터넷뱅킹 실시간 이체 | 필수 |
| **가상계좌** | 입금 확인 후 처리 | 필수 |
| **네이버페이** | 네이버페이 간편결제 | 높음 |
| **카카오페이** | 카카오페이 간편결제 | 높음 |
| **페이코** | PAYCO 간편결제 | 중간 |
| **토스페이** | 토스 간편결제 | 중간 |

### 5.2 PG 연동 구조

```
후원자 → 결제 UI → PG사 (이노페이/헥토파이낸셜/KG이니시스 등)
       → 결제 승인 → 웹훅(Webhook) 수신
       → 납입 레코드 생성 (상태: 수입대기)
       → ERP 연동 API 호출 가능 상태
```

#### 결제 관련 자동화

- **CMS 출금 자동처리**: 매월 출금일에 금융결제원 CMS 청구 자동 실행
- **CMS 동의 ARS 녹취**: 신규 CMS 신청 시 ARS 자동 증빙 연동
- **카드 정기결제 자동처리**: 매월 자동 청구, 실패 시 재시도 로직
- **미납 자동 감지**: 출금 실패 건 자동 분류 및 알림

### 5.3 결제 실패 처리 정책

| 실패 유형 | 처리 방법 |
|----------|---------|
| CMS 잔액 부족 | D+1, D+3, D+7 재시도 후 미납 처리 |
| 카드 한도 초과 | 즉시 알림 SMS 발송, 결제수단 변경 유도 |
| 카드 만료 | 만료 1개월 전 갱신 안내 발송 |
| 계좌 해지 | 관리자 알림 + 후원자 SMS 통보 |
| 3회 연속 실패 | 스케줄 일시정지, 관리자 상담 알림 |

## 6. 기능 요구사항 - 사용자(후원자) 기능

### 6.1 후원 신청 플로우

**왜 필요한가**: 후원 신청 과정이 복잡하면 이탈률이 급증한다. Bloomerang 연구에 따르면, 3단계 이내 완료 시 전환율이 2배 이상 높다.

```
단계 1: 후원 유형 선택 (정기/일시/정기+일시)
단계 2: 금액 선택 또는 직접 입력
단계 3: 개인정보 입력 (이름, 연락처, 이메일, 주소)
단계 4: 결제 수단 선택 및 결제 진행
단계 5: 완료 화면 (감사 메시지, 후원증서 즉시 발급)
```

#### 본인인증 정책

- 실명 인증: 기부금 영수증 발급 요건 충족을 위해 CI/DI 기반 인증 권장
- 미인증 허용: 소액 일시후원은 인증 없이 가능 (정책 설정 가능)
- 외국인: 여권번호 기반 인증 경로 제공

### 6.2 후원자 마이페이지

| 기능 | 상세 |
|------|------|
| **납부 내역 조회** | 월별/연도별 후원 이력, 엑셀 다운로드 |

- **통합 주소록**: 모든 인적자원을 하나의 데이터베이스에서 조회

- **활동 이력 자동 저장**: 주요 정보 변경 시 변경 전/후 내역 자동 기록
- **파일 첨부**: 문서, 녹취파일, 스캔이미지, 사진, 동영상 (암호화 저장)

## 8. 기능 요구사항 - NPO ERP 연동

### 8.1 연동 목적

**왜 필요한가**: 후원금 수납과 회계 처리가 분리되면 재정 투명성이 저하되고 결산 작업이 중복된다. 도너스의 ERP 연동 가이드를 참고하여 표준 API 인터페이스를 설계한다.

### 8.2 연동 데이터 모델

도너스 API 분석을 기반으로 다음 계층 구조를 채택한다.

```
구성원 (Member)
  └── 약정 (Promise / Subscription)
        ├── 약정A (일시납)
        │     └── 납입1 (Payment)
        └── 약정B (정기납)
              ├── 납입1 (월 1회 자동 생성)
              ├── 납입2
              └── ...
```

#### 납입(Payment) 핵심 필드

| 필드명 | 설명 | 타입 |
|--------|------|------|
| `paymentIdx` | 납입 고유 인덱스 | integer |
| `paymentCode` | 납입 코드 | string |
| `memberIdx` | 구성원 인덱스 | integer |
| `memberCode` | 구성원 코드 | string |
| `name` | 구성원명 | string |
| `promiseIdx` | 약정 인덱스 | integer |
| `promiseCode` | 약정 코드 | string |
| `productCode` | 모금상품(캠페인) 코드 | string |
| `purposeCode` | 용도 코드 (기금 분류) | string |
| `payPrice` | 납입금액 | integer |
| `payDate` | 납입일 (결제 예정일) | date |
| `depositDate` | 입금일 (실제 입금 확인일) | date |
| `incomeStatus` | 수입상태 (수입대기/진행/완료/제외) | enum |
| `payMethodType` | 자산구분 (CMS/카드/계좌이체 등) | string |
| `payMethodCode` | 납입방법 코드 | string |
| `bankAccountCode` | 납입계좌 코드 | string |

### 8.3 ERP 연동 API 명세

#### API 1: 납입정보 조회 (List Payments)

```
GET /api/v1/payments
Authorization: Bearer {api_key}
```

**Request Parameters:**

| 파라미터 | 설명 | 필수 |
|---------|------|------|
| `startDate` | 조회 시작일 (YYYY-MM-DD) | 필수 |
| `endDate` | 조회 종료일 (YYYY-MM-DD) | 필수 |
| `memberIdx` | 특정 구성원 납입만 조회 | 선택 |
| `dateType` | 날짜 기준 (payDate: 납입일 / depositDate: 입금일) | 선택 |
| `incomeStatus` | 수입상태 필터 (1:수입대기, 2:수입진행, 3:수입완료) | 선택 |
| `count` | 페이지당 결과 수 (기본 10, 최대 1년) | 선택 |

**Response (JSON):**

```json
{
  "startDate": "2026-01-01",
  "endDate": "2026-01-31",
  "incomeStatus": 1,
  "totalCount": 454,
  "payments": [
    {
      "paymentIdx": 22919805,
      "paymentCode": "P26000012",
      "memberIdx": 623769,
      "memberCode": "M001234",
      "name": "홍길동",
      "promiseIdx": 45678,
      "promiseCode": "PR240001",
      "productCode": "CAMP_2026_01",
      "purposeCode": "CHILD_CARE",
      "payPrice": 30000,
      "payDate": "2026-01-25",
      "depositDate": null,
      "incomeStatus": 1,
      "payMethodType": "CMS",
      "payMethodCode": "AUTO_TRANSFER",
      "bankAccountCode": "BA_001"
    }
  ]
}
```

#### API 2: 납입상태 변경 (Change Status of Payments)

```
PATCH /api/v1/payments/{paymentIdx}/status
Authorization: Bearer {api_key}
```

**Request Body:**

```json
{
  "incomeStatus": 3,
  "depositDate": "2026-01-26",
  "memo": "수입결의서 처리 완료"
}
```

**Response:**

```json
{
  "success": true,
  "paymentIdx": 22919805,
  "previousStatus": 1,
  "currentStatus": 3,
  "updatedAt": "2026-01-26T14:30:00Z"
}
```

#### API 3: Webhook (Push 방식 - 실시간 연동 추가)

```
POST {ERP_WEBHOOK_URL}/donation/received
```

도너스와 달리, 본 시스템은 ERP가 주기적으로 조회하는 Pull 방식 외에 **실시간 Push(Webhook)** 방식도 지원하여 ERP 처리 지연을 최소화한다.

```json
{
  "event": "payment.created",
  "paymentIdx": 22919806,
  "memberCode": "M001234",
  "payPrice": 30000,
  "incomeStatus": "PENDING",
  "occurredAt": "2026-01-26T15:00:00Z"
}
```

### 8.4 연동 대상 ERP 시스템

| ERP 유형 | 연동 방식 | 우선순위 |
|---------|---------|---------|
| **비영리 회계 ERP (국내)** | REST API (본 명세 기반) | 필수 |
| **MRM (휴먼소프트웨어)** | RESTful Open API 연동 | 높음 |
| **아이엠데이터 ERP** | REST API | 중간 |
| **엑셀/CSV 내보내기** | 수동 연동 (API 미지원 ERP) | 필수 |
| **국세청 홈택스** | 연말정산 간소화 전산매체 | 필수 |

### 10.2 영수증 정책

| 상황 | 처리 정책 |
|------|---------|
| 주민번호 없음 | 영수증 미발급 안내, 정보 추가 입력 유도 |

### 10.3 데이터 정책

| 상황 | 처리 정책 |
|------|---------|
| 후원자 탈퇴 | 법정 보관 기간(5년) 경과 후 물리적 삭제, 그 전까지 비식별화 |

```
├── Campaign (캠페인)
│     ├── DonationType (후원 유형 설정)
│     └── DonationPage (랜딩 페이지)
│
└── Member (구성원/후원자)
      ├── MemberType (후원자/봉사자/잠재후원자 등)
      ├── Contact (연락처)
      ├── PaymentMethod (결제 수단 - 카드/CMS 계좌)
      └── Promise (약정)
            ├── PromiseType (정기/일시)
            ├── ProductCode → Campaign 연결
            └── Payment (납입)
                  ├── incomeStatus (수입대기/진행/완료/제외)
                  ├── DonationReceipt (기부금영수증)
                  └── ERPLedger (ERP 수입결의)
```

### 11.2 주요 상태 다이어그램 (납입)

```
[결제 시작] → [수입대기(PENDING)]
                    ↓
              [ERP 조회]
                    ↓
            [수입진행(PROCESSING)] ← ERP가 처리 중임을 표시
                    ↓
            [수입완료(COMPLETED)] ← ERP 수입결의 완료
                    
[특수] → [수입제외(EXCLUDED)] ← 환불/취소/오류 건
```

## 13. 구현 우선순위 로드맵

**Deloitte Quick Wins 접근법** 기반으로 단계적 구현

### Phase 1 — MVP (0~3개월): 빠른 성과 창출

- [ ] 캠페인 랜딩페이지 생성 (노코드 편집기)
- [ ] 정기/일시 후원 결제 연동 (카드, CMS, 네이버페이, 카카오페이)
- [ ] 결제 완료 자동 감사 SMS/이메일 발송
- [ ] 기본 관리자 대시보드 (후원자 목록, 납부 현황)
- [ ] 기부금 영수증 자동 발급
- [ ] 후원자 마이페이지 (납부 내역, 정기후원 관리)

### Phase 2 — Core (3~6개월): 운영 효율화

- [ ] ERP 연동 API (납입정보 조회 + 납입상태 변경)
- [ ] ERP Webhook 실시간 Push 연동
- [ ] CRM 통합 (후원자/봉사자/잠재후원자 통합 관리)
- [ ] 자동화 메시징 (출금 실패, 미납 관리, 생일 등)
- [ ] 권한 관리 (본부/지부/담당자)
- [ ] 보고서 및 통계 대시보드

### Phase 3 — Advanced (6~12개월): 전략적 확장

- [ ] AI 기반 후원자 이탈 예측 (Churn Prediction)
- [ ] P2P 펀드레이징 (후원자가 직접 모금 페이지 생성)
- [ ] 국세청 연말정산 간소화 자동 연계
- [ ] 국내외 아동결연관리 (SCRM) 모듈
- [ ] 배분관리 모듈 (기금 배분 추적)
- [ ] 모바일 앱 (관리자용 + 후원자용)
- [ ] 오픈 API 공개 (외부 연동 지원)
